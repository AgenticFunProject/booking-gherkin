# Executable Gherkin Runner Design

This document defines the intended runner for the Booking black-box contract in
`test/features/*.feature`. It is a design artifact only; executable runner code
is a follow-up bead after the wording and CI entrypoint are ready.

## Goals

- Discover every `test/features/*.feature` file without listing them in Java
  code.
- Start the Booking Service in a controlled `test,local` runtime with local
  Schedule, Quote, and Equipment stubs.
- Send real HTTP requests to the public Booking API and actuator/documentation
  routes.
- Assert only externally visible status codes, headers, content types, and JSON
  fields.
- Keep step glue and fixtures separate from the contract prose so replacement
  implementations can run the same features.

## Tooling

Use a Java 21 test-scope runner so the contract can share the repository Maven
toolchain without coupling scenarios to Spring internals:

| Concern | Choice |
| --- | --- |
| Gherkin engine | Cucumber JVM on the JUnit Platform |
| HTTP client | RestAssured or JDK `HttpClient`; prefer RestAssured for JSON path assertions |
| JSON parsing | Jackson `JsonNode` for reusable field, table, and regex assertions |
| JWT creation | JJWT in test scope, matching the service test JWT configuration |
| Process control | A small Java fixture that starts/stops the service as an external process |
| Maven entrypoint | A dedicated profile or plugin execution such as `./mvnw test -Pcontract` |

The runner package should live outside production packages, for example
`src/contractTest/java/com/cargo/booking/contract`. Glue must not import
controllers, services, repositories, entities, mappers, or Spring test helpers.
Shared values may be duplicated as contract fixtures instead of reusing
application classes.

## Feature Discovery

The JUnit Platform suite should select `test/features` as the feature root and
use Cucumber tags for partial runs:

| Selector | Purpose |
| --- | --- |
| `test/features/**/*.feature` | Full contract discovery |
| `@public-routes`, `@booking-create`, `@auth`, etc. | Focused local debugging |
| `not @wip` | Default exclusion for incomplete future scenarios |

Feature discovery belongs in runner configuration, not in the feature files.
Adding a new `.feature` file should be enough for the next full contract run to
pick it up.

## Service Startup

Start the Booking Service once per contract run unless a scenario explicitly
requires a restart. The launcher should:

1. Reserve a free HTTP port and set `server.port`.
2. Start the app with `spring.profiles.active=test,local`.
3. Enable test resources on the app classpath when using `spring-boot:run`, or
   provide equivalent temporary config when running the packaged jar.
4. Set `app.security.enabled=false` by default for local-mode feature files.
5. Override to `app.security.enabled=true` for security-tagged scenarios.
6. Set the JWT issuer, audience, and secret to the test values used by the
   token fixture.
7. Wait for `/actuator/health` before executing the first step.

The runner must treat the service as a remote HTTP target after startup. It must
not call Spring beans, repositories, or test utility classes to create state or
read assertions.

Persistence scenarios need controlled restarts. For those scenarios, stop only
the service process, leave the same database running, restart the service with
the same configuration, and assert visibility through public API calls.

## Scenario State

Use one scenario-scoped context object. It should be reset before every
scenario and hold only data observed or generated by the runner:

| State | Purpose |
| --- | --- |
| Base URL | Target host and port for HTTP calls |
| Headers | Authorization, `X-Request-ID`, content type, and per-step overrides |
| Latest response | Status, headers, raw body, parsed JSON |
| Latest booking id | Follow-up route calls after a create or fixture setup step |
| Latest booking reference | Reference-based read/list assertions |
| Named bookings | Cross-customer ownership scenarios with more than one booking |
| Named customers | Stable customer IDs used by auth and list scenarios |
| Active token | Current caller token, if any |

The state model should support named aliases like `customer 3002's booking`,
`latest booking`, and `existing pending booking`. Aliases are local to the
scenario and must not leak to later scenarios.

## Reset Strategy

Isolation should favor deterministic data over execution order assumptions:

- Generate unique customer IDs per scenario when wording allows it.
- Reset headers, latest response, saved IDs, and tokens before every scenario.
- Reset durable booking data before every scenario that does not explicitly test
  restart-safe persistence.
- For reset, prefer recreating the runner-owned PostgreSQL schema or database.
  Direct database cleanup is runner infrastructure only; feature steps and
  assertions must continue to observe behavior through HTTP.
- For persistence-runtime scenarios, reset before the scenario, then keep the
  same database through the in-scenario service restart.

If a future implementation exposes a test-only HTTP reset endpoint, the runner
may use it through the same public target URL, but the contract must not require
that endpoint from production deployments.

## JWT Fixture

The token fixture should generate HS256 JWTs with:

| Claim/config | Default |
| --- | --- |
| Issuer | `test-issuer` |
| Audience | `equipments-service` |
| Secret | `test-secret-key-that-is-at-least-256-bits-long` |
| Expiration | One hour in the future |
| Role claims | `roles` array for `CUSTOMER`, `SERVICE`, and `OPERATOR` |
| Admin compatibility | Accepted admin role representation from the service tests |
| Customer identity | `customerId` or `customer_id` claim depending on the scenario |

The fixture must also generate expired tokens, wrong issuer tokens, wrong
audience tokens, malformed bearer values, and tokens signed with the wrong key.
Steps only attach these tokens as HTTP headers; they must not inspect service
security internals.

## Reusable Step Phrase Families

The glue should support the wording already used by the feature files rather
than forcing feature rewrites.

| Phrase family | Examples |
| --- | --- |
| Runtime mode | `the Booking Service is running with the "local" profile`, `security is disabled`, `security enabled` |
| Local stubs | `the local schedule client stub accepts schedule validations`, `schedule, quote, and equipment dependencies accept the default booking fixture` |
| Generic HTTP | `I send a GET request to "/path"`, `a client requests booking "..." with header "X-Request-ID" equal to "..."` |
| Create fixture | `a client creates a booking with this request:`, `a booking was created with this request:`, `customer 3001 has an existing "PENDING" booking` |
| Saved values | `the latest booking id is saved for this scenario`, `the latest booking reference is saved for this scenario` |
| Retrieval/listing | `retrieves the latest booking by id`, `retrieves the latest booking by booking reference`, `lists bookings with customerId 3001` |
| Lifecycle | `confirms`, `starts`, `completes`, `cancels`, `attempts to confirm/start/complete/cancel` |
| Auth callers | `a customer JWT contains role "CUSTOMER" and customerId 3001`, `a caller JWT contains role "SERVICE"` |
| Status/assertions | `the response status is 201`, `the response status must be 403`, `the response status should be 200 or a redirect status` |
| JSON assertions | `the response body has "status" equal to "PENDING"`, `contains a field named "timestamp"`, `describes a validation failure for "equipment"` |
| Table assertions | `the response body should contain:`, `equipment containing exactly:`, `every returned booking must have customerId 3001` |

Where files use `is`, `must be`, and `should be` variants for the same
assertion, glue should map them to one implementation.

## JSON Assertion Rules

Assertions should operate on parsed JSON and stable public fields:

- Support dotted paths such as `customer.email` and array checks such as
  `equipment`.
- Compare numeric values as JSON numbers, not as strings.
- Match timestamps by ISO-8601 parseability and UTC normalization.
- Match generated references by regex and current UTC year.
- Treat field order as irrelevant.
- For validation errors, assert the field name and status/error shape without
  requiring exact violation array ordering unless the scenario says so.

## Black-Box Boundary

The runner may control process startup, database lifecycle, and JWT generation
as test harness responsibilities. It must not make contract assertions through
implementation-specific APIs.

Allowed in runner infrastructure:

- Starting and stopping a Booking process.
- Allocating ports and database names.
- Cleaning the runner-owned database between scenarios.
- Generating request JSON and JWTs.
- Polling public health routes for readiness.

Not allowed in step glue or assertions:

- Calling `BookingService`, repositories, mappers, controllers, or authorizers.
- Reading JPA entities or database rows to decide whether a scenario passed.
- Depending on table names, SQL, Flyway migration contents, bean names, package
  names, logs, or exact stack traces.
- Requiring real Schedule, Quote, or Equipment HTTP endpoints before their
  owning teams publish stable contracts.

## CI Hook

CI and local development use a single Maven entrypoint that runs after the
normal unit/integration/e2e gates:

```bash
./mvnw test -Pcontract
```

The current profile discovers every `test/features/*.feature` file and validates
the executable contract corpus with the `test,local` runtime configuration
properties that the future HTTP glue will use. When the full Cucumber HTTP
runner is added, it should keep this command stable and move the assertions from
corpus validation to black-box HTTP scenario execution.
